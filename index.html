The Big Picture

AI agents are learning to buy things on behalf of humans. Stripe, Coinbase, and Skyfire are building the payment plumbing (how agents pay). Nobody has built the trust layer (should agents pay this particular site?).

OpenTrustSeal fills that gap. One API call. Under 200 milliseconds. The agent gets a signed, verifiable answer: this site is safe, proceed with caution, or do not transact.

Foundation

Key Concepts Explained

These are the technical building blocks behind OpenTrustSeal. You do not need to become an engineer, but understanding these concepts will let you speak confidently about the product.

RFC stands for "Request for Comments." It is the standard format used to propose and define internet protocols since the 1960s. HTTP (how web pages load), SMTP (how email works), and TLS (how encryption works) were all defined as RFCs.

When we write our protocol spec in RFC style, we are signaling to the technical community: "This is a serious standard, not a product pitch." It means we define every field, every response code, every edge case. Engineers can read it and implement it without guessing.

Why it matters for marketing: When you tell a CTO "we have an RFC-style spec," they immediately understand that this is an open, implementable standard. It is the difference between saying "we have a product" and "we have a protocol." Protocols become infrastructure. Products get replaced.

The W3C (World Wide Web Consortium) is the organization that defines web standards. They define how HTML works, how CSS works, and how web browsers should behave. They are the closest thing the internet has to a governing body for technical standards.

Verifiable Credentials (VCs) are a W3C standard for digital certificates that can be independently verified without calling the original issuer. Think of it like a digital notarized document.

A physical analogy: Your driver's license was issued by the DMV. A bartender can look at it and trust it without calling the DMV because it has security features (holograms, specific format) that are extremely hard to forge. Verifiable Credentials work the same way, but digitally and with math instead of holograms.

Our trust token IS a Verifiable Credential. This means any system that already understands VCs can read our tokens without custom integration. We are building on an existing standard rather than inventing our own format.

Why it matters for marketing: "Our trust tokens follow the W3C Verifiable Credentials standard" tells technical buyers we are not reinventing the wheel. We follow the same standard that governments and banks are adopting for digital identity.

Cryptographic signing is how you prove a document has not been tampered with and was created by who it claims to be from.

Imagine you write a letter, seal it with a unique wax stamp that only you own, and mail it. The recipient can see the unbroken seal and know: (1) this letter really came from you, and (2) nobody opened or changed it in transit. Digital signatures do the same thing with mathematics.

Ed25519 is a specific signing algorithm. There are many algorithms to choose from (RSA, ECDSA, Ed25519, etc.). Ed25519 is the modern choice because it is fast (signing takes microseconds), the signatures are small (64 bytes), the keys are small, and it has no known vulnerabilities. It was designed by Daniel Bernstein, one of the most respected cryptographers alive.

How it works in OTT: We have a private key (secret, locked in a hardware vault) and a public key (published openly). When we score a website, we sign the trust token with our private key. Anyone with our public key can verify the signature is authentic. If even a single character of the token were changed after signing, the verification would fail.

Why it matters for marketing: "Cryptographically signed trust tokens" means nobody can forge a trust score. Not the site owner, not a middleman, not even a hacker who intercepts the token. Only OpenTrustSeal can produce valid tokens, and everyone can verify them. This is the core trust guarantee of the entire product.

DID stands for Decentralized Identifier. It is a W3C standard for creating digital identities that do not depend on any single authority.

Think of it this way: your email address (you at a provider) depends on that provider. If the provider disappears, your identity disappears. A DID is like an email address that you own permanently, independent of any company.

Our DID is did:web:opentrustseal.com. This is our digital identity. Attached to this identity is our public signing key. When an agent receives a trust token signed by us, it checks our DID to find our public key and verify the signature.

Key management is the practice of securely storing, rotating, and publishing cryptographic keys. Our private keys live in an HSM (Hardware Security Module), which is a physical device designed so that keys can never be extracted from it, even if someone physically steals the device.

Why it matters for marketing: DID-based key management means agents worldwide can verify our tokens without any prior relationship with us. They look up our DID, find our public key, and verify. No API calls, no accounts, no contracts needed for basic verification. This is what makes the protocol truly open.

A .well-known URI is a standardized location on any website where specific information can be found. It is defined by an internet standard (RFC 8615) that says: if you want to publish machine-readable metadata about your site, put it in the /.well-known/ directory.

Examples that already exist:

  • /.well-known/security.txt tells security researchers how to report vulnerabilities
  • /.well-known/apple-app-site-association tells iOS which app to open for a website
  • /.well-known/openid-configuration tells login systems how to authenticate

Our addition: /.well-known/ots.json tells AI agents whether this site is trustworthy. Sites publish their signed trust token here. Any agent visiting any website can check this one URL and get an instant trust assessment.

Why it matters for marketing: We are not asking sites to install complex software. We are asking them to host a single JSON file at a standard location. That is as simple as web integration gets. "Add one file to your site" is a much easier sell than "integrate our SDK."

An HSM (Hardware Security Module) is a physical, tamper-resistant device that stores cryptographic keys. Banks, certificate authorities, and governments use HSMs to protect their most sensitive secrets.

The key property: the private key is generated inside the HSM and never leaves it. When we need to sign a trust token, we send the data into the HSM, it signs it internally, and sends back only the signature. Even our own engineers cannot extract the key. If someone physically tampers with the device, it destroys the key automatically.

Cloud providers offer HSM-as-a-service (AWS CloudHSM costs roughly $1.50/hour). We do not need to own physical hardware, but we get the same security guarantees that banks use.

Why it matters for marketing: "Bank-grade key security" is not a buzzword when you actually use HSMs. This is the same technology protecting Visa's transaction signing keys. When a payment rail asks "how do you protect your signing keys?" the answer "HSM with FIPS 140-2 Level 3 certification" ends the conversation.

KYC stands for "Know Your Customer." It is a regulatory and business practice where a company verifies the real-world identity of its clients. Banks do KYC when you open an account (checking your ID, verifying your address, confirming your Social Security number).

In our context, KYC means verifying the identity of the people behind a website. When a site owner wants to move from the free tier to a paid verification tier, we verify:

  • Enhanced ($29/mo): Business name, address, and phone are real
  • KYC-Verified ($99/mo): Government ID, business documents, bank account, video call
  • Enterprise ($499/mo): All of the above plus on-site audit and continuous monitoring

Why this is the monetization engine: New sites automatically score low because they have no history. The only way to quickly raise their score is to verify their identity. Sites that want AI agents to send them money are highly motivated to prove they are legitimate. We charge for this verification, and the incentives are perfectly aligned: higher verification means lower fraud risk for everyone.

An API (Application Programming Interface) is a way for one computer to talk to another over the internet. When an AI agent wants to check if a website is trustworthy, it sends a request to our API (like visiting a URL) and gets back a structured answer (the trust token).

Think of it as a drive-through window. The agent pulls up, asks a question ("Is merchant.com safe?"), and gets an answer in a standard format. The agent does not need to know how we computed the answer. It just needs to know where to ask and how to read the response.

An SDK (Software Development Kit) is a pre-built library that makes calling an API easier. Instead of an agent developer writing 20 lines of code to call our API, parse the response, check the signature, and handle errors, they install our SDK and write 3 lines of code. SDKs exist for different programming languages (we will offer Python and JavaScript SDKs).

Why it matters for marketing: "One API call" and "three lines of code" are the phrases that sell to technical decision makers. Integration cost is their biggest concern. The easier the integration, the faster the adoption.

JSON (JavaScript Object Notation) is the most common format for structured data on the web. It looks like this:

"domain": "example.com",
"trustScore": 82,
"recommendation": "PROCEED"

It is readable by both humans and machines. Every programming language can read and write JSON.

JSON-LD ("JSON for Linked Data") adds a layer of meaning to JSON. The @context field at the top of our trust token says "here is where to find the definition of what each field means." This allows any system in the world to understand our data without prior coordination.

Think of JSON as a form with labeled fields. JSON-LD adds a link to the instruction manual that explains what each label means.

Why it matters for marketing: JSON-LD is what Google uses for rich search results and what schema.org uses for product data. By adopting JSON-LD, our trust tokens are automatically compatible with a massive existing infrastructure of tools and standards.

The Protocol

The protocol defines how trust information flows from our servers to AI agents. There are two paths an agent can take.

Two Ways Agents Verify Trust

Path A: Direct Discovery

The agent visits the site and reads the trust token file directly. No API call to us needed. The agent verifies the cryptographic signature on its own.

Best for: Speed. Privacy. Decentralization.

Path B: API Call

The agent calls our API and we return the latest trust data. This path supports bulk checks, fresher data, and richer metadata.

Best for: Freshness. Bulk operations. Enriched data.

How a Verification Happens (Step by Step)

Agent wants to pay merchant.com
→
Agent calls OTT API
→
OTT returns score + recommendation
→
Agent decides: proceed, pause, or refuse

The entire check takes under 200 milliseconds. For context, a human blink takes 300 to 400 milliseconds. The agent gets its answer before you finish blinking.

What the Trust Token Looks Like

This is a simplified view of what an agent receives when it checks a domain:

The token also includes a full breakdown of each signal category (domain age, SSL, DNS security, content quality, reputation, and identity verification) so agents can make nuanced decisions rather than relying on a single number.

Why the Signature Matters

Without signatures: A scam site could create a fake trust token claiming a score of 99. Agents would have no way to know it was forged. It would be like anyone being able to print their own driver's license.

With signatures: Only OpenTrustSeal can produce valid tokens. A scam site can put any file at /.well-known/ots.json, but the agent will check the cryptographic signature and immediately reject it. It is mathematically impossible to forge.

System Architecture

The system has 10 major components. Here is what each one does, explained without jargon.

CDN (Content Delivery Network)

A network of servers worldwide that cache our data close to users. When an agent in Tokyo checks a site, the response comes from a server in Tokyo, not from our headquarters. This is how we hit sub-200ms response times globally.

API Gateway

The front door. Every request enters here first. It checks who is asking (authentication), whether they have asked too many times (rate limiting), and routes the request to the right service. Think of it as the receptionist.

Verify Service

Handles the core question: "Is this domain trustworthy?" It checks the cache first (fast), falls back to the database if needed (slower), and returns the trust token. This is the most-used component.

Registration Service

Handles new site sign-ups. When a site owner registers, this service generates a verification code and confirms they actually own the domain (via DNS record or file upload).

Signal Collectors

Six independent workers that gather trust data: domain age, SSL status, DNS configuration, website content, reputation (malware/phishing checks), and identity verification status. They run on schedules and feed data into the scoring engine.

Scoring Engine

Takes all six signals, applies weights (reputation and identity count the most), applies the tier ceiling, checks for critical flags, and produces the final trust score. Pure math, no human judgment.

Signing Service

The most secure component. Takes a computed score and stamps it with our cryptographic signature through the HSM. This signature is what makes tokens unforgeable. Private keys never leave the hardware vault.

Cache (Redis)

An in-memory data store that holds every active trust token. Because it runs in memory (RAM), lookups take 1 to 2 milliseconds compared to 15ms for a database query. 99% of verification requests are served from cache.

Database (PostgreSQL)

The permanent record. Stores all domains, signals, scores, KYC submissions, API keys, and audit logs. The cache is fast but temporary. The database is the source of truth.

Message Queue

The internal mail system. When a signal collector finishes a check, it drops a message in the queue. The scoring engine picks it up and recalculates. When a score changes, another message triggers re-signing. This keeps everything loosely connected and independently scalable.

Why This Architecture Matters for the Pitch

When a CTO at Stripe or Coinbase evaluates a partner, they look at the architecture. "Can this handle our volume?" is always the first question. By designing every component to scale independently, we can answer: "We handle 60 queries per minute today and 6,000 per minute tomorrow without rebuilding anything. We just add more instances."

The phrase to use: "horizontally scalable at every layer." That means we grow by adding more machines, not by buying bigger machines. It is the standard that enterprise buyers expect.

The Neutrality Advantage

A trust authority must be neutral to be credible. Stripe cannot build a trust layer because they would be judging which merchants agents should pay, and they process the payments. That is a conflict of interest. Same problem for Coinbase and Google.

The analogy: SSL certificates come from independent certificate authorities (DigiCert, Let's Encrypt), not from Chrome or Safari. Browsers trust CAs because CAs have no stake in which sites you visit. OTT is the CA model applied to agent payments.

The one-liner for your pitch deck:

"Every agent payment company is building the rails. We built the road signs."

Why Speed Matters

The defense against big players copying this is not the code (the protocol is open). The defense is the dataset. Every site that completes KYC verification through us adds to a registry that no competitor can replicate without doing the same work from scratch. At 10K verified sites, we have a useful dataset. At 100K, we have a moat. At 1M, we are the standard.

First mover wins in trust infrastructure because trust compounds. The longer a site has been verified, the more data we have on it, the more accurate the score, the more agents trust our system, the more sites want to join.

Quick Glossary

Terms you will encounter in meetings and documents, defined in plain language.

API
A way for one computer to ask another computer a question and get a structured answer, over the internet.
SDK
A pre-built code library that makes it easy for developers to call an API. "Three lines of code" instead of twenty.
CDN
A network of servers worldwide that cache data close to users for speed. Your Netflix streams from a local CDN node, not from California.
HSM
A tamper-proof hardware device that stores cryptographic keys. The key is generated inside and never leaves. Banks and governments use these.
Ed25519
A modern, fast, secure algorithm for creating digital signatures. The mathematical guarantee that a trust token was not forged.
DID
Decentralized Identifier. A digital identity that is not controlled by any single company. Our DID tells agents where to find our public key.
W3C
World Wide Web Consortium. The organization that defines web standards (HTML, CSS, Verifiable Credentials). The governing body of the web.
Verifiable Credential
A W3C standard for digital documents that can be verified without calling the issuer. Our trust tokens follow this format.
RFC
Request for Comments. The standard format for internet protocol specifications. HTTP, email, and TLS are all defined as RFCs.
JSON
A simple, universal format for structured data. Every programming language can read it. Our trust tokens are JSON documents.
KYC
Know Your Customer. The process of verifying the real-world identity of a person or business. Banks do this when you open an account.
.well-known
A standard directory on any website where machine-readable metadata lives. Our trust token sits at /.well-known/ots.json.
WHOIS
A public database that records who registered a domain name and when. We check it for domain age.
TLS/SSL
The encryption that makes websites show a padlock icon. It protects data in transit between a browser and a server.
DNS
Domain Name System. The internet's phone book that translates "google.com" into a server address. We check its security configuration.
SPF/DMARC
Email security standards that prevent email spoofing. Their presence indicates a domain owner cares about security configuration.
DNSSEC
DNS Security Extensions. Adds cryptographic verification to DNS lookups. Prevents attackers from redirecting a domain to a fake server.
Horizontal Scaling
Growing by adding more machines rather than making one machine bigger. This is how Netflix and Google handle growth. It is what enterprise buyers expect.
Cache
A fast, temporary copy of data stored in memory (RAM). 99% of our verification requests are answered from cache in under 10 milliseconds.
SRI
Subresource Integrity. A browser feature that verifies a script file has not been tampered with by checking its hash before running it.
