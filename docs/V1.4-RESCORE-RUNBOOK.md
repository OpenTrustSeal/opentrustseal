# v1.4 Rescore Runbook

Step-by-step for re-scoring the OpenTrustSeal registry with the v1.4 scoring model. Run after the Codex fix bundle deploys (which introduces the v1.4 model along with confidence + cautionReason fields on the signed payload). Assumes the post-seed-merge DB is on the API box at `/opt/opentrusttoken/data/ott.db`.

## What rescore.py does

The script reads every row from `raw_signals` (the per-domain signal snapshots captured at crawl time), re-runs the current `compute_score` / `compute_confidence` / `compute_caution_reason` functions on that data, and writes updated rows to `scored_results` + `score_history`. No network, no WHOIS re-query, no Playwright -- purely a scoring-model replay against stored evidence.

This is the payoff of keeping raw signals separate from scores. Scoring-model changes are cheap to apply; recrawls are not.

## When to run

- After a scoring-model version bump (v1.3 → v1.4 here).
- After a bugfix to any `compute_*` function.
- After a post-seed-merge import (merge brings in 100K+ new raw_signals rows that need scoring).
- NOT for a collector fix (if a collector changes how raw data is captured, you need a re-crawl, not a rescore).

## Pre-flight

Before you touch anything:

```bash
ssh root@206.189.65.177

# 1. Snapshot the DB. Cheap insurance.
cp /opt/opentrusttoken/data/ott.db /opt/opentrusttoken/data/ott.db.pre-v1.4-rescore.$(date +%Y%m%d-%H%M%S)

# 2. Confirm the current scoring model version in the code
grep '^SCORING_MODEL' /opt/opentrusttoken/app/scoring.py
# Should print: SCORING_MODEL = "ots-v1.4-weights"

# 3. Confirm raw_signals row count matches your expectation
sqlite3 /opt/opentrusttoken/data/ott.db "SELECT COUNT(*) FROM raw_signals;"
# Pre-seed-merge: ~3900 rows. Post-seed-merge: ~100K+ depending on merge.

# 4. Confirm scored_results row count
sqlite3 /opt/opentrusttoken/data/ott.db "SELECT COUNT(*) FROM scored_results;"
```

## Step 1 -- Dry run

Never skip this. A silent bug in the rescore path mutates 1000+ rows in one pass.

```bash
cd /opt/opentrusttoken
sudo -u ott /opt/opentrusttoken/venv/bin/python3 rescore.py --dry-run 2>&1 | tee /tmp/rescore-dry-$(date +%Y%m%d-%H%M%S).log
```

Expected output (pre-seed, 1231 rows):

```
DRY-RUN: scored_results will NOT be written. Remove --dry-run to apply.
...
Rescored 1231 domains
Score range: 10--88 | Average: 68.8 (was 68.8)
...
(dry-run, nothing written)
```

What to look for in the log:

- **Score deltas.** The log prints per-domain `old_score -> new_score` for every changed row. After v1.4 specifically, the Tranco top-100 consensus tier should move Amazon/Google/Stripe/etc from mid-70s into the 80-90 band. Spot-check 5-10 well-known top-Tranco domains.
- **No crashes.** Script should exit 0. If it crashes mid-way, inspect the last processed domain and its raw_signals row for a data-shape surprise.
- **Reasonable distribution shift.** Average score change should be a few points at most. A 20+ point swing means the scoring model regressed or a signal is being double-counted.

## Step 2 -- Spot-check the deltas

Pipe the log through a quick filter to find the biggest movers:

```bash
# Show domains that moved more than 10 points in either direction
grep -E "\->" /tmp/rescore-dry-*.log | awk -F'->' '{
    split($1, a, " "); split($2, b, " ");
    old = a[length(a)]; new = b[1];
    delta = new - old;
    if (delta > 10 || delta < -10) print $0;
}' | head -30
```

For v1.4 specifically, expected top movers (into the 80+ band):
- amazon.com, google.com, microsoft.com, apple.com, cloudflare.com: +5 to +10
- Top-100 Tranco retailers: +2 to +7
- No domain should drop more than 5 points. If one does, the v1.4 logic has a regression on that evidence shape. Fix before committing.

## Step 3 -- Apply for real

After the dry-run log looks clean:

```bash
cd /opt/opentrusttoken
sudo -u ott /opt/opentrusttoken/venv/bin/python3 rescore.py 2>&1 | tee /tmp/rescore-live-$(date +%Y%m%d-%H%M%S).log
```

The live pass takes roughly 1 second per 1000 domains (it is pure CPU: read raw, compute, write signed result, no network). For 100K rows expect ~1-2 minutes.

Real-time monitor from another shell:

```bash
watch -n 5 'sqlite3 /opt/opentrusttoken/data/ott.db "SELECT COUNT(*), scoring_model FROM scored_results GROUP BY scoring_model;"'
```

You should see rows move from `ots-v1.3-weights` to `ots-v1.4-weights` as the pass progresses.

## Step 4 -- Post-flight verification

```bash
# 4a. All rows now on v1.4?
sqlite3 /opt/opentrusttoken/data/ott.db "SELECT scoring_model, COUNT(*) FROM scored_results GROUP BY scoring_model;"
# Expected: only ots-v1.4-weights

# 4b. Distribution of confidence values (should be populated on every row)
sqlite3 /opt/opentrusttoken/data/ott.db \
  "SELECT json_extract(response_json, '$.confidence') AS confidence, COUNT(*)
   FROM scored_results GROUP BY confidence;"
# Expected: mix of "high"/"medium"/"low", zero "unknown".

# 4c. cautionReason populated only on CAUTION rows
sqlite3 /opt/opentrusttoken/data/ott.db \
  "SELECT json_extract(response_json, '$.cautionReason') AS reason, COUNT(*)
   FROM scored_results WHERE recommendation='CAUTION' GROUP BY reason;"
# Expected: incomplete_evidence / weak_signals / new_domain / infrastructure
# None should be empty or NULL after v1.4 rescore.

# 4d. Hit the stats endpoint and confirm dataset view is populated
curl -s https://api.opentrustseal.com/v1/stats/dataset | python3 -m json.tool
# Expected: byConfidence map populated, byCautionReason map populated,
# agentSafePercent reflects the new v1.4 distribution.
```

## Rollback plan

If the rescore produced obviously-wrong results:

```bash
ssh root@206.189.65.177
systemctl stop opentrustseal      # freeze the API
cp /opt/opentrusttoken/data/ott.db.pre-v1.4-rescore.<timestamp> /opt/opentrusttoken/data/ott.db
systemctl start opentrustseal
```

Raw signals are not touched by rescore, so even without the snapshot backup you could roll scoring_model back to v1.3 by reverting `scoring.py` and re-running `rescore.py`. The snapshot is belt-and-suspenders for sub-5-second recovery.

## Common gotchas

- **`compute_confidence` needs `domain_age_days`.** Pre-Codex-fix code called it without that argument. If you see a TypeError about missing positional argument, you are running old code. Check `from app.scoring import compute_confidence` in rescore.py signature matches.
- **`_unscorable` key missing on old rows.** Raw_signals rows crawled before the Codex fix bundle won't have `_unscorable` in their content payload. `rescore.py` handles this with `raw.get("content", {}).get("_unscorable", False)` defaulting False. Do not "fix" this to raise on missing -- it would break rescore on the 1231 pre-fix rows.
- **Per-process DBs after seed merge.** If you rescore a merged DB that combined 18 seed DBs, some domains might have multiple raw_signals rows from different seeds. `rescore.py` reads each, but `scored_results` has PRIMARY KEY on domain so only the latest scoring wins. This is fine, just be aware.
- **Signing key must be present.** `rescore.py` re-signs every response. If `/opt/opentrusttoken/keys/` is missing, the script errors before touching the DB. Run `python3 -c "from app.signing import ensure_keys; ensure_keys()"` first if needed.

## When to re-run

Re-run `rescore.py` whenever:

- `SCORING_MODEL` version string in `scoring.py` changes.
- You merge new seed DBs via `merge_db.py` (merge brings in raw_signals rows with no corresponding scored_results; rescore populates them).
- `compute_confidence` or `compute_caution_reason` changes in a way that would alter output on existing data.

Do NOT re-run on every deploy. Only on scoring-model changes.
